v0.3.9: section chunking for L2 truncation on outlier-sized sections (Task #34)

Some projects (e.g., ibd_phage_targeting) have REPORT or RESEARCH_PLAN
sections that exceed any reasonable max_tokens budget when fed to L2.
v0.1.10 added length-aware retry; v0.1.11 raised the ceiling to 64K;
both still leave outlier sections truncated. v0.3.9 adds the missing
piece: split the input section into smaller chunks before extraction.

## Design

**chunking.chunk_section()** — three-tier boundary cascade:
  1. Subheading boundaries (### or #### within the section)
  2. Paragraph boundaries (\\n\\n) when no subheading break is available
  3. Character-fallback for paragraphs that exceed threshold themselves

Default threshold: 12_000 chars (~3K tokens). Tunable per call.
No overlap initially — content-preserving splits, dedup at section
level by (entity_kind, canonical_id, surface_form).

Chunks under threshold pass through as a single chunk with chunk_id=None.
This is the cache-preservation lever (see below).

**ExtractionCache** — `get()` and `put()` accept an optional `chunk_id`:
  - chunk_id=None → cache key is byte-identical to v0.3.7. Every existing
    cache row from v0.1.x – v0.3.7 (~25K on Adam's hub) remains valid.
  - chunk_id="i/N" → mixed into content_hash; each chunk gets its own row.

**UniversalExtractor.extract()** — refactored to factor per-call work into
`_extract_chunk()`. Section under threshold takes the existing path
(unchunked, chunk_id=None, single LLM call). Section over threshold
iterates chunks, each cached/LLM-called/parsed independently, then
merged + deduped at the section level.

The crucial property: if your cache predates v0.3.9 and you didn't
have any oversized sections, NO cache rows invalidate on upgrade.
Every section that was a hit before is still a hit.

## Tests (11 new, 142/142 passing)

`tests/unit/test_v039_chunking.py`:
- chunk_section produces 1 chunk under threshold (chunk_id=None — the
  cache-preservation lever)
- chunk_section produces N chunks over threshold, all with valid chunk_id
- chunk_section respects threshold across packing
- subheading-aware split when ### present
- paragraph fallback when no subheadings
- character fallback for giant single paragraphs
- concatenation recovers content modulo blank-piece pruning
- cache key for chunk_id=None vs chunk_id="0/2" are distinct
- chunked round-trip (3 chunks, 3 cache rows, all retrievable)
- **test_extract_small_section_uses_unchunked_cache_key** — the
  cache-preservation gate: pre-seed a row with chunk_id=None as v0.3.7
  would have, run extract() on a small section, assert ZERO LLM calls.
  This test fails loudly if anyone refactors away from chunk_id=None
  for unchunked sections.
- test_extract_large_section_chunks_and_caches_per_chunk — cold scan
  produces N LLM calls + N cache rows; warm rescan produces 0 LLM calls
  (all hits); mention dedup collapses identical extractions across chunks
  to a single mention.

## Memory

Added `feedback_cache_key_chunked_only_when_chunked.md` capturing the
generalizable pattern: when adding an optional dimension to a cache key,
make the new component opt-in so the default behavior produces the
pre-change key shape. Applies to any expensive-to-recompute keyed store
(LLM caches, embedding caches, materialized views).

## Backward compatibility

- Default scan behavior unchanged for sections under 12K chars (the vast
  majority).
- Sections over 12K chars that were previously cached as a single
  truncated entry will be detected as a `finish_reason='length'` cache
  miss (existing v0.1.8 behavior) and re-extracted via chunking on next
  scan.
- No schema migration; the cache table's columns are unchanged.

## What this fixes for Adam

The IBD project's oversized REPORT/RESEARCH_PLAN sections — and any
similar outliers across the corpus — will now extract cleanly into
multiple cache rows + multiple LLM calls per section, each well within
the 64K output budget. Mentions and drift candidates are merged at the
section level and deduped, so downstream warehouse + dashboard panels
see one coherent set of mentions per section regardless of chunk count.
