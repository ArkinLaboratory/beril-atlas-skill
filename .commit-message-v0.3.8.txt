v0.3.8: Untracked-projects panel + persistent-cache CLI flags

Two fixes that were queued from the v0.3.6 diagnostic loop on Adam's hub:
the untracked-projects panel makes the data-omission visible without
silently fabricating dates, and --cache-path / --seed-cache-from let
users keep cache hits across runs without copying files manually.

## Fix 1 — Untracked-projects panel (Task #40)

Symptom: Q5 of the v0.3.6 diagnostic surfaced 4 projects on Adam's hub
with extracted conclusions but NULL effective_completion_date AND NULL
completion_date — 251 conclusions silently invisible to every trend
panel because trend SQL filters on
COALESCE(effective_completion_date, completion_date) IS NOT NULL.
Affected projects: field_vs_lab_fitness,
enigma_contamination_functional_potential, costly_dispensable_genes,
amr_strain_variation. Each has L1+L2 extraction but no Revision History
entry in their RESEARCH_PLAN/REPORT, so completion_date never gets
populated.

Fix: new Act-3 panel (panel-untracked-projects) listing these projects
with their last_touched, conclusion count, file count, size, and
authors. Sortable+filterable table primitive, click-to-open project
drawer.

Surface-only design (deliberate). Atlas does NOT silently fall back to
last_touched in enrich_projects — that would conflate filesystem mtimes
with intentional research dates and lose this exact diagnostic. The
panel-claim explicitly tells the user the fix is data-side: add a
Revision History entry to the project's RESEARCH_PLAN/REPORT.

Empty-state: if every project has a date, the panel renders a "No
dropouts" message instead of the table. Tested both states.

## Fix 2 — --cache-path and --seed-cache-from CLI flags (Task #15)

Symptom: per memory `reference_atlas_cache_per_outputs_root.md`, the L2
extraction cache lived inside --outputs-root pre-v0.3.8. Fresh
timestamped run = empty cache = full re-extract (~$5–10 + 45 minutes
cold). Users wanting persistent cache across runs had to copy
extraction_cache.duckdb manually between outputs_root timestamps.

Fix: two new flags on `beril-atlas scan`:

  --cache-path PATH
    Override the default outputs_root/extraction_cache.duckdb. Point at
    a stable location (e.g., ~/.beril-atlas/cache/extraction_cache.duckdb)
    and cache hits persist across runs even when --outputs-root is a
    fresh timestamp.

  --seed-cache-from PATH
    Copy a prior extraction_cache.duckdb into the destination cache
    before extraction starts. Useful when you want a fresh
    --outputs-root but warm-cache from a prior scan. Refuses to
    overwrite an existing destination cache (safety: delete first or
    omit the flag to use the existing cache as-is).

Implementation: extracted the resolution logic into
scan._resolve_cache_path() so it's testable without spinning up the
LLM client. Manifest's files_generated now records the actual cache
path (was hardcoded to outputs_root).

## Backward compatibility

- Default scan behavior unchanged: cache still lives at
  outputs_root/extraction_cache.duckdb if --cache-path is not set.
- Manifest files_generated entry now uses the actual cache path
  (previously hardcoded). Run-comparison tooling should already handle
  cache_path key in the l2_extraction summary block.
