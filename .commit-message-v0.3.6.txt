v0.3.6: April-trend visibility + discoveries-drawer becomes a sortable/filterable table

Three bundled fixes responding to live observations on Adam's hub against
the 58-project corpus on 2026-04-27.

## Fix 1 — Monthly tick labels on cumulative-trend x-axes

Symptom: Adam reported "we are still seeing nothing after April 1st" on
every trend panel. Root cause was NOT data: the v0.3.4 warehouse had 5
April projects with effective_completion_date set, 28 April revisions,
987 April conclusions properly bucketed in the panel-side SQL. The bug
was Plotly's auto-tick algorithm: with a 3-month data range, it chose
weekly tick labels (Feb 1, Feb 8, ..., Mar 29) and never labeled "Apr
2026," so the rightmost datapoint sat in unlabeled space and the eye
read it as "nothing past April 1." The line genuinely extended to the
April point — there was just no axis label to anchor the read.

Fix: every cumulative-trend chart x-axis now uses tickmode='array' with
explicit tickvals/ticktext computed from the data months. April 2026
gets its own labeled tick, no exceptions. Affected:
- render_topic_trends_panel (organism/method/database/function trends)
- render_discoveries_timeline
- render_negative_result_rate_panel (Monthly view)
- render_positive_result_rate_panel (Monthly view)
- render_revision_kinds_chart (Revisions stacked bar)

Diagnostic regret: I burned three release cycles attacking the wrong
layer (v0.3.3 added a COALESCE fallback, v0.3.4 fixed a populator
column-count crash) before screenshots surfaced the real cause. Memory
entry feedback_screenshot_before_layer_diagnosis.md captures the lesson:
when data-layer diagnostics return clean, that IS the signal to ask the
user for a screenshot of the surface, not to keep proposing data-layer
fixes.

## Fix 2 — Discoveries drawer becomes a sortable/filterable table

Symptom: Adam clicked an April datapoint on the discoveries timeline
and saw 20 samples ALL from enigma_sso_asv_ecology, even though 5 April
projects contributed hundreds of conclusions. Root cause:
fetch_claims_by_month_and_type used ORDER BY (project_id, mention_id)
+ LIMIT rn<=20 within each (month, claim_type) bucket. The
alphabetically-first project with ≥20 claims swallowed the entire
20-slot bucket. Buckets that should have shown 5 projects showed 1.

Fix: pivot from "SQL pre-decides which 4 of each project to show" to
"give the user all the claims and let them filter." The drawer now
renders a sortable+filterable HTML table with 4 columns: Project,
Source (doc § section), Claim, Verbatim quote. User filters to a single
project to focus, sorts by source_section to group claims from the same
section, etc. SQL changes to per_project_cap=20 (replaces the buggy
overall limit_per_cell), so each contributing project surfaces up to 20
of its claims and total payload stays bounded.

The sortable/filterable JS primitive was already in the dashboard from
v0.1.10 but ran exactly once on page load (IIFE). For the drawer to
work — table is injected dynamically when a chart point is clicked — the
init was refactored into window.wireSortableTable / wireFilterableTable
/ wireTablesIn, idempotent via data-*-wired attributes so re-clicks
don't double-wire.

Distinct-project count is now surfaced in the drawer header
("12 claims from 5 projects; capped at 20 per project") so coverage is
visible at a glance before any filtering.

## Fix 3 — Untracked-projects awareness (data only; panel deferred)

Q5 of the diagnostic surfaced 4 projects with conclusion-bearing
extraction but NULL effective_completion_date — that's 251 conclusions
silently invisible to every trend panel. v0.3.6 doesn't yet add a panel
listing them; that's intentional (Task #40) — the surface-only fix
("show me what's missing") is more informative than a fall-through
("invent a date so nothing's missing"). Inventing dates would lose the
diagnostic. Holding for design pass.
