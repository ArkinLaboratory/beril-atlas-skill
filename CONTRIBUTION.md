# Contribution + promotion flow + leak-test checklist

**Date:** 2026-04-24
**Status:** DRAFT — Task #5 deliverable. Awaits Adam review.

## When this flow applies

Not at v0.1 ship. The seed vocab in the first release of `ArkinLaboratory/beril-atlas-skill`
is Adam's curated spike vocab, one-way ship-only. This flow activates when:

- A friend's install has run one or more scans.
- Drift review in that install has surfaced terms that are methodologically
  portable (generic methods, databases, organisms, functions) rather than
  deployment-specific jargon.
- The install's operator wants to propose those terms for inclusion in the
  shipped vocab so everyone benefits.

The first real contribution cycle may not happen until v0.2 or later.
This doc specs the mechanism so we don't improvise when the moment arrives.

## Context: BERIL /submit already syncs projects across installs

Confirmed by Adam 2026-04-24. BERIL's `/submit` command propagates projects
across installs that have enabled sync. Implications for this flow:

- **The "cross-install" dimension is implicit in the corpus.** Installs with
  sync enabled see overlapping project sets. An atlas `project_count` within
  one install is already a count over the union of local projects plus
  synced-in projects. Cross-install aggregation at the vocab layer is less
  necessary than it would be for installs with fully-private corpora.
- **Contributions are about operator judgment, not corpus aggregation.**
  Two operators scanning similar synced corpora may independently converge
  on proposing the same terms — that's signal. But the aggregation logic
  still lives in the human-review step, not in code.
- **Leak tests apply uniformly.** A drift candidate derived from a synced-in
  project goes through the same sanitization gates as one derived from a
  local project. The operator may not know which projects came from sync;
  gate 2 (no project identifiers) strips origin regardless.

This strengthens the case for deferring cross-install aggregation tooling
(see Gate 5 discussion below).

## What can flow back (recap from bucket audit)

Three types of BIDIR contribution, in order of expected volume:

1. **Vocab promotions** — new canonical terms or synonyms for existing
   categories. Highest volume, most mechanical, most leak-prone.
2. **Methodology corrections** — runtime gotchas, risk-register additions,
   scoring-weight refinements. Low volume, high variance.
3. **Synthetic fixture refreshes** — maintainer-internal, not driven by
   install-side contributions. Mentioned for completeness.

No other categories flow back in v0.1. Extraction cache, warehouse outputs,
run manifests, and project-derived data are strictly LOCAL — see bucket
audit section "Runtime artifacts not currently in tree."

## Contribution flow — install side

### Where contributions are staged

Per LAYOUT.md:

```
<BERIL_ROOT>/.claude/skills/beril-atlas/contrib/
├── README.md                         # written on install-skill first run
├── vocab-promotions-<date>.yaml      # one file per drift-review session
├── methodology-<date>.md             # one file per contribution
└── ...
```

`contrib/` is install-local. `beril-atlas install-skill` creates the dir
but never overwrites files inside. Users commit to contribution via an
explicit action — copy/attach the relevant file to a PR or email.

### Vocab-promotion generation

CLI tool: `beril-atlas drift-promote` (added as a new CLI subcommand
post-v0.1).

Workflow:

1. User runs their scans normally. Drift candidates accumulate in the
   warehouse `drift_candidates` table.
2. User runs `beril-atlas drift-promote --review`. Tool walks candidates,
   groups by (category, candidate_term), computes frequency stats:
   - `project_count` — in how many projects this term appeared
   - `mention_count` — total mentions
   - `first_seen_at`, `last_seen_at`
3. Tool filters to candidates passing the **term-frequency floor**
   (leak-test #5 below). Default floor: `project_count >= 3`.
4. Tool presents each passing candidate to the user via AskUserQuestion:
   - Choices: `Propose for upstream`, `Keep local only`, `Discard`, `Defer`.
   - User sees: candidate_term, proposed canonical form, proposed category,
     proposed synonyms (if any), frequency stats. **No project IDs shown** —
     only counts. **No quoted extraction text shown** — those live in the
     cache only.
5. For each "Propose for upstream" candidate, tool appends a sanitized
   entry to `contrib/vocab-promotions-<date>.yaml`.
6. For each "Keep local only," tool appends to `vocab-local/<category>.local.yaml`.
7. "Discard" drops the candidate (written to `drift_decisions` table with
   `decision=reject`).
8. "Defer" leaves the candidate in `drift_candidates` for the next review.

### vocab-promotion file format

```yaml
# Generated by beril-atlas drift-promote on 2026-04-24T15:00:00Z
# beril-atlas-skill version: 0.1.0
# Install marker (opaque install ID; does NOT identify the install's corpus):
#   install_id: a7c3d4e1       # hash of skill-dir path, for maintainer dedup
#
# Review this file before submitting:
#   - Confirm no canonical term or synonym contains text specific to your
#     projects (project slugs, author names, unique descriptors).
#   - Confirm the category assignments are correct.
#   - Delete entries you don't want to propose.
#
# Submit via PR against ArkinLaboratory/beril-atlas-skill or email to aparkin@lbl.gov.

promotions:
  - category: methods
    canonical: transposon_insertion_sequencing
    synonyms:
      - "tn-seq"
      - "transposon sequencing"
      - "tnseq"
    stats:
      project_count: 7
      mention_count: 34
      first_seen: 2026-03-15
      last_seen: 2026-04-22
    note: |
      (Optional user-authored note. Free-form but must not contain project
      names, author names, or quoted extraction text. Empty by default.)

  - category: databases
    canonical: mibig
    synonyms:
      - "minimum information about a biosynthetic gene cluster"
      - "MIBiG database"
    stats:
      project_count: 4
      mention_count: 11
      first_seen: 2026-04-10
      last_seen: 2026-04-20
    note: ""
```

### methodology-correction format

For runtime gotchas or methodology refinements (volume will be low; keep
format simple):

```markdown
# methodology-2026-04-24.md

**Proposed change:** (one-line summary)

**Type:** runtime-gotcha | risk-register-addition | scoring-refinement | docs-correction

**Category:** (which shipped doc this affects — e.g., references/risks-and-caveats.md)

**Proposed text:** (the new content, ready to paste)

**Rationale:** (why this matters — what you observed, what broke, what would
fail silently without this)

**Reproduction:** (if applicable — minimal conditions to reproduce the
underlying observation; MUST NOT contain project-specific data)

**Install context:** package version, approximate BERIL deployment size,
platform (macOS/Linux/Windows). NOT project identifiers.
```

## Contribution flow — maintainer side

### Intake

Users submit contributions via one of:

1. **PR** against `ArkinLaboratory/beril-atlas-skill` — `contrib/vocab-promotions-<date>.yaml`
   pasted into a `contrib-pending/<date>-<userhandle>.yaml` file in the PR.
2. **Email** to `aparkin@lbl.gov` — attached YAML, maintainer manually files
   to `contrib-pending/`.

Once filed, the leak-test pipeline runs against the pending file (below).
No changes ship without passing all 7 gates.

### Release cadence — Option B (release when meaningful, cost visible)

Decided 2026-04-24. Release when a contribution set carries meaningful
methodological benefit, not on a fixed time schedule. Small, frequent
releases are preferred over large quarterly batches, for these reasons:

- Contributors see their terms shipped in weeks, not quarters — the
  feedback loop that sustains contribution effort.
- Smaller releases are easier to differential-test (Gate 7). Large batches
  make it hard to isolate which specific term caused an unexpected delta.
- Transparent versioning — "v0.3 added 4 databases, v0.4 added 7 methods."
  Users know exactly what changed.

### The cache-invalidation cost is per-rescan, not per-release

Each extracted mention is cached keyed by
`(content_hash, prompt_version, vocab_version, model_id)`. Bumping a vocab
file's version invalidates cached rows. At current scale (53 projects,
~1200 extractable sections) that's ~5M tokens and ~45 min wall-clock for a
full re-scan.

**But users control when they pay that cost.** After a release:

- `pipx upgrade beril-atlas-skill` — pulls the new package; nothing in
  the user's BERIL deployment changes yet.
- `beril-atlas install-skill .` — copies new `vocab-shipped/`; existing
  scans/warehouses/caches remain valid. Next scan uses new vocab and
  incurs re-extraction cost only for sections that hit the new vocab.
- Full re-scan — user chooses to trigger a fresh scan to re-apply new vocab
  to their entire corpus. Full cost.

So release cadence does NOT force cost on users. Users upgrade on their
own cadence with full knowledge of what they're committing to.

### Release notes must make the cost visible

Every release documents:

```
v0.2.0 — 2026-07-15

Vocab additions:
  methods: 4 new canonical terms (transposon_insertion_sequencing,
    dual_rna_seq, ...)
  databases: 2 new canonical terms (mibig, kbase_refdata)

Cache impact:
  If extraction_cache vocab_version is monolithic (single hash across
  all vocab files): any re-scan re-extracts everything. At 53-project
  scale this is ~5M tokens / ~45 min.

  [If per-category, only methods + databases rows invalidate; organisms
   and other categories remain cached. Task #6 verifies which.]

Recommended action:
  Upgrade + install-skill at your convenience. Full re-scan only needed
  if you want the new vocab applied to your existing corpus. Otherwise,
  new vocab applies to your next routine scan.
```

### vocab_version cache-key granularity — verified 2026-04-24

**Finding:** `vocab_version` in the extraction-cache key is constructed
per-category but used monolithically.

In `engine/extractors/universal.py:143-145`:
```python
vocab_version_str = ":".join(
    f"{k}={vv.version}" for k, vv in sorted(self.vocabularies.items())
)
```

So the string looks like `"databases=1:journals=1:methods=1:organisms=1:question_types=1"`.
Bumping `methods` to v2 changes the string to
`"databases=1:journals=1:methods=2:organisms=1:question_types=1"` — which
invalidates the cache for every section, not just sections touching
methods.

**Why:** the UniversalExtractor does one LLM call per section, covering
all vocab kinds together. The cost win of joint extraction (~6x vs
per-category) is architecturally tied to joint cache keys. Per-category
cache invalidation would require per-category LLM calls and a 6x cost
regression on normal scans.

**Implication:** the release-cadence cost framing remains correct as
written above. Bumping any vocab category triggers full re-extraction for
users who choose to re-scan. Release notes must continue to make this
cost visible. Option B release cadence stands.

**Future-proofing:** if we ever decide the cache-invalidation cost is too
high, the refactor path is to split the extractor into per-kind LLM calls.
That's a significant redesign (~2x-6x LLM cost on every scan, not just at
vocab-bump time). Not worth it for v0.x. Revisit only if real friends-tier
data shows users skipping upgrades specifically because of re-scan cost.

### Promotion into shipped vocab

Mechanical steps:

1. Verify all 7 leak tests pass (see checklist below).
2. Open `skill/vocab-shipped/<category>.v{N+1}.yaml` as a copy of v{N}.
3. Append accepted entries from pending YAML. Preserve existing canonical
   order; new entries go at the end or in alpha-sorted position.
4. Bump `version: {N+1}` in the file's frontmatter/header.
5. Update `vocab-reference.md` in `references/` if the new entry is
   methodologically significant (e.g., adds a whole new category of method).
6. Run the differential fixture test (leak test #7) — does the shipped
   fixture render correctly with the new vocab?
7. Commit: `vocab(<category>): promote N terms from contrib-<date>`.
8. Delete the pending file from `contrib-pending/`.
9. Tag the release.

## Leak-test checklist (7 gates)

Every contribution passes ALL seven before entering shipped. Gates 1–5
are mechanical (can be scripted as a pre-commit hook on `contrib-pending/`);
gates 6 and 7 are human discipline.

### Gate 1 — No raw extraction text

Contribution files must be summary/categorical. Specifically:
- No quoted sentences from documents.
- No LLM-generated summaries derived from specific corpora.
- Notes are free-form prose but must be about the *concept* (what the
  term means) not about *specific instances* (what projects used it).

**Automated check:** scan contribution YAMLs + markdowns for
- Any string > 160 chars (likely a quote)
- Any string containing triple-backticks or double-quoted multi-sentence
  blocks
- Any string matching LLM-output markers (`"response":`, `"content":`)

**Fail mode:** hard reject, ask submitter to restate concept-level.

### Gate 2 — No project identifiers

Scan for:
- Regex `\b(projects?/)?([a-z][a-z0-9_-]{3,})\b` where the matched slug
  looks project-name-shaped (snake_case, 4+ chars, has underscore). False
  positives are tolerated — better to over-reject than leak.
- Explicit list of known BERIL project slugs from the shipped fixture's
  provenance block (updated each release).
- Author names: scan for `Dehal`, `Arkin`, and any ORCID-pattern
  `\d{4}-\d{4}-\d{4}-\d{3}[\dX]`.

**Automated check:** `beril-atlas lint-contrib <file>` runs the scan.

**Fail mode:** hard reject, flag offending lines.

### Gate 3 — No citation-graph fragments

Contribution files must not contain `(src, dst)` edge rows or reuse-edge
descriptions that name specific projects.

**Automated check:** reject any key named `src_project_id`, `dst_project_id`,
`reuse_edge`, `citation_edge`, etc. in contrib YAMLs.

**Fail mode:** hard reject.

### Gate 4 — No cache rows

Never ship `.duckdb` files, `.parquet` files, or any binary in contrib/.
Never ship `extraction_cache` content in any form.

**Automated check:** `.gitignore` in `ArkinLaboratory/beril-atlas-skill` excludes
`contrib/**/*.duckdb`, `contrib/**/*.parquet`. Pre-commit hook enforces
"contrib/**" contains only `.yaml`, `.md`, `.txt` files.

**Fail mode:** hard reject at git commit time.

### Gate 5 — Term-frequency floor

Candidate must pass a frequency threshold before it's contributed. Default:
`project_count >= 3`. The thought is: a term appearing in only 1–2 projects
is likely deployment-specific jargon or a transient LLM hallucination, not
a methodologically portable term.

**Automated check:** `beril-atlas drift-promote --review` filters
sub-threshold candidates out before presenting to the user. User cannot
accidentally propose a term that appeared only once. Threshold tunable
via `--project-count-floor` flag for power users.

**Cross-install floor — deferred.** BERIL `/submit` already provides the
cross-install signal implicitly (see BERIL sync context above). Maintainer
doing hand-aggregation across 2–3 submitted YAMLs is faster than designing
speculative tooling. Explicit trigger to revisit: **when contribution
volume exceeds 5 submissions per release**, or friends start reporting
that hand-curation introduces noise. At that point, build
`beril-atlas maintain aggregate <contrib-pending/>` to compute cross-install
frequency and pre-filter the promotion queue.

**Fail mode:** candidate not shown in the promotion UI.

### Gate 6 — Human diff gate

Before merging any contribution, maintainer (Adam, eventually BERIL team)
reviews a side-by-side diff of `skill/vocab-shipped/<category>.v{N}.yaml`
vs proposed `.v{N+1}.yaml`. Purpose: catch anything gates 1–5 missed.

**No automation.** Explicitly a human step. Review takes ~10–30 minutes per
release for a typical batch.

**Fail mode:** reject specific entries with feedback. Accepted entries go
in; rejected entries are noted in the PR/email with reason.

### Gate 7 — Differential fixture test

After merging promotions into shipped vocab, run the synthetic fixture
scan (using `beril-atlas fixture-regen`) and compare output against the
previous release's fixture output. Expected deltas:

- New canonical terms appear in vocab count rows
- Frequency of already-present terms may shift slightly (cache
  invalidation effect)
- Dashboard panel counts should move by predictable small amounts

**Automated check:** `beril-atlas fixture-test --against v{N}` produces
a diff report. Unexpected changes (e.g., a new project suddenly appearing,
a metric swinging by orders of magnitude) indicate a bug in the vocab
merge — not a leak, but a correctness issue.

**Fail mode:** block release; investigate.

## Summary of what flows where

| Artifact | Where it lives | Who writes it | Who reads it | Leak gates applied |
| --- | --- | --- | --- | --- |
| Seed vocab | `skill/vocab-shipped/*.v1.yaml` | Adam (v0.1 seed) | runtime at scan time | gates 1–7 before shipping v1 |
| User vocab additions | `<BERIL>/.claude/skills/beril-atlas/vocab-local/` | user hand-edits | runtime overlay | none (local only, never ships) |
| Drift candidates | warehouse table | engine | `drift-promote` CLI | none (install-local) |
| Promotion candidates | `contrib/vocab-promotions-<date>.yaml` | `drift-promote` CLI | submitter, then maintainer | gates 1–5 at generation; 6 at merge |
| Pending contributions | `contrib-pending/` in shipped repo | maintainer on intake | maintainer at review time | gates 1–5 re-validated, 6 applied |
| Shipped vocab v{N+1} | `skill/vocab-shipped/*.v{N+1}.yaml` | maintainer at release | runtime at scan time | gate 7 before release |
| Extraction cache | `state/extraction_cache.duckdb` | engine | engine | NEVER crosses boundary |
| Warehouse | `~/.beril-atlas/runs/<ts>/` | engine | user | NEVER crosses boundary |

## Decisions (resolved 2026-04-24)

1. **Submission channel — email** initially. Adam will email friends when
   the first release is ready; friends email back contribution YAMLs.
   Once the repo goes public or friends-volume justifies, PR-via-GitHub
   becomes primary. No PR template to maintain at v0.1.
2. **Cross-install aggregation tooling — defer.** BERIL `/submit` already
   provides implicit cross-install corpus overlap. Hand-aggregation by
   maintainer is the right size until submission volume exceeds ~5 per
   release. Explicit trigger documented in Gate 5.
3. **`drift-promote` default-shows the "propose for upstream" choice.**
   Default-hide would discourage well-meaning contribution; good filter
   defaults on the gates keep noise out.
4. **Install ID as path-hash — keep.** Worth the minor complexity. Lets
   maintainer dedup "these 3 contributions came from the same install"
   without identifying who. Privacy-preserving.
5. **Release cadence — Option B** (release when meaningful, make cost
   visible in release notes). Documented in "Release cadence" section
   above.
