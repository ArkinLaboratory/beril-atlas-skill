v0.3.11: Sidebar nav state + scroll positioning fixes (Task #46)

Three independent bugs reported on the live hub dashboard 2026-04-28:
"sidebar nav not maintaining state, not jumping to exact location."
Each is a small JS/CSS fix; together they restore expected click +
scroll + multi-section behavior.

## Bug 1 — Hash clobbers act selection (state bug)

Symptom: clicking any sidebar link to a sub-panel (e.g.,
#panel-database-matrix) flashed the correct tab briefly then bounced
back to Act 0.

Root cause: the click handler relied on the browser's native
anchor-jump (no e.preventDefault). After activate(act5) ran, the
browser updated location.hash to #panel-foo, which fired
hashchange → activateFromHash → "doesn't start with 'act'" branch →
fell back to Act 0.

Fix:
1. activateFromHash now maps non-act hashes to their enclosing
   section.act (mirrors the click handler's logic).
2. Click handler does e.preventDefault() and uses
   history.replaceState instead of letting the browser update the
   hash (replaceState does NOT trigger hashchange — clean).

## Bug 2 — IntersectionObserver collapses manually-opened sections

Symptom: open sidebar section 5 to see its TOC entries, scroll
through panels in act 3, section 5 silently slams shut.

Root cause: the IO unconditionally set
  s.open = (s.dataset.act === newAct)
on every cross-act scroll, closing every section except the new
active one. Comment claimed "manual state respected within an act"
but only WITHIN one act — crossing acts nuked manual state.

Fix: additive auto-open. Now does
  if (s.dataset.act === newAct && !s.open) s.open = true;
which opens the new act's section but never closes others.

Cost: across a long browsing session, sections accumulate open. User
can manually close any that get cluttered. Strictly better than
aggressive auto-collapse.

## Bug 3 — Anchor jump lands target panel under sticky tab-nav

Symptom: clicking any sidebar link scrolled to the right panel but
the panel header sat behind the sticky nav, hiding the title row.

Root cause: nav.tab-nav is position:sticky; top:0 with ~50–60px
height after padding/border. .panel had scroll-margin-top:1rem (16px)
which is less than the sticky-nav height. Anchor jump landed the
panel top 16px below the viewport top → behind the nav.

Fix: html { scroll-padding-top:5rem; } on the root scrolling
container. Modern browsers honor this for all anchor jumps; cleaner
than per-element scroll-margin-top tuning. 5rem clears a wrapped
tab-nav on narrow viewports.

Also: click handler now does explicit
  requestAnimationFrame(() =>
    target.scrollIntoView({behavior:'smooth', block:'start'}))
after activate() runs, so layout settles before the scroll fires —
combined with scroll-padding-top:5rem this lands the panel cleanly
below the nav with smooth animation.

## Tests (4 new, 152/152 passing)

`tests/unit/test_v0311_sidebar_nav.py` — source-text regressions on
the rendered HTML. The runtime behavior is exercised in the browser,
but each pattern silently breaks user-visible nav if removed:

- test_v0311_html_has_scroll_padding_top — assertion against the
  CSS rule that prevents Bug 3.
- test_v0311_activate_from_hash_maps_to_enclosing_act — asserts
  the closest('section.act') lookup is present (prevents Bug 1).
- test_v0311_intersection_observer_is_additive — asserts the
  guarded `&& !s.open` pattern is present AND the unconditional
  assignment idiom (with semicolon, distinguishing code from
  doc-comments) is absent (prevents Bug 2).
- test_v0311_click_handler_prevents_default_and_uses_replace_state —
  preventDefault, history.replaceState, scrollIntoView with
  block:'start' all present.

## Backward compatibility

- No data-layer changes. Warehouse, cache, and CLI unaffected.
- All previous panels render identically; only nav behavior changes.
- The CSS scroll-padding-top:5rem applies to ANY anchor jump on the
  page (including non-sidebar in-page links). The 5rem offset is
  cosmetically uniform — should not break any existing in-page navigation.
