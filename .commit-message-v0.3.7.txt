v0.3.7: discoveries-drawer table column-width fix

Layout-only patch on top of v0.3.6.

Symptom (caught 2026-04-27 on Adam's hub against the live 4684-conclusion
warehouse): the v0.3.6 discoveries drawer rendered the sortable/filterable
table at full panel width but with no column-width constraints, so the
longest cell in any column determined that column's width. The Verbatim
Quote column held strings up to ~400 chars; combined with the 14% nominal
project-id width getting stretched by long IDs like
"enigma_sso_asv_ecology", the quote column bled off the panel's right
edge with no scrollbar and no ellipsis. Mid-word truncations like
'"would capture the 56% of rea…' were the visible failure.

Fix: table-layout:fixed + a <colgroup> with explicit per-column widths
(project 14%, source 16%, claim 38%, verbatim quote 32%) + word-wrap +
overflow-wrap on every long-text cell. Removed white-space:nowrap from
project/source columns since fixed table layout doesn't need it and the
no-wrap forced overflow under fixed layout.

Tests:
- 122/122 unit tests pass.
- Existing test_v036_discoveries_drawer_renders_sortable_filterable_table
  extended to assert table-layout:fixed, <colgroup>, and word-wrap presence.
  Without these, the v0.3.6 layout regression returns silently.

No data-layer changes. fetch_claims_by_month_and_type, the SQL,
per_project_cap=20, and the round-robin click flow are all unchanged from
v0.3.6.
