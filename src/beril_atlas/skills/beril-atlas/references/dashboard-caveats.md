# BERIL Atlas — Risks, Caveats, and Honest Framing Register

**Purpose.** Every claim in the dashboard lives or dies against one of these
caveats. This is the single source of truth for "what can the atlas honestly
assert?" The dashboard's closing-caveats act links here; panel-level caveats
reference specific entries by number.

**Maintenance.** Update when a new interpretive pitfall is discovered.
Archive (don't delete) resolved caveats so historical dashboards remain
interpretable.

---

## A · Stage-of-system caveats (most consequential for current readers)

### A1. BERIL is pre-beta; measured dynamics reflect an early-stage system

BERIL is not production research infrastructure yet. Early-stage dynamics
(rapid churn, bursts of activity, narrow author base) are normal for
pre-beta software and do not generalize to how the system will operate at
scale.

**What this means for dashboard readers:** do not extrapolate current
per-project rates, per-author productivity, or topic distribution as
steady-state. Trends observed in the bring-up period of a deployment
describe *system bring-up*, not sustained operation.

### A2. Author-domination is common in early-stage deployments

When one author is on a majority of projects in the corpus, the "author
interaction graph" and "research lines" narratives primarily trace that
author's trajectory with small hand-offs to collaborators. This is the
expected shape of a deployment in its bring-up phase.

**What this means:** collaboration metrics are diagnostic of a solo
power-user pattern rather than of a multi-team system. Cross-author edge
classification (per risk D4b) partially corrects for this in `influence`
and `integration` scores — self-follow-on by a deep-diver author is
reported as its own axis rather than as external influence. The "how
work flows across authors" panels become meaningful as the user base
broadens.

### A3. Corpus is in active expansion

A BERIL deployment early in its life is in a phase of active author and
project recruitment. Snapshots taken close together in time may look like
substantially different systems as new authors arrive, new project clusters
form, and cross-author handoffs become substantive for the first time.

**What this means:** every run is timestamped in the manifest. Readers
should compare **trajectories** across runs rather than interpret
single-snapshot metrics as steady state. Baseline-period metrics are NOT
valid predictors of post-expansion behavior — that's the whole point of
the atlas being an instrument that measures change, not a photograph.

## B · Data-collection caveats

### B1. Corpus is filesystem-time, not organic git history

BERIL projects are filesystem directories, not per-project git repositories.
When projects arrive in a corpus, their filesystem `mtime` values may all
equal the deployment-drop time. Project "dates" therefore come from internal
Revision History v1 timestamps, not filesystem events. This is fine for
ordering but means filesystem `mtime` is not a usable temporal signal.

### B2. No project-level git repositories

BERIL projects are not git repos. Per-project velocity metrics
(commits/week, author attribution via blame) are unavailable. Author
attribution comes from parsing the Authors section of canonical docs.

### B3. Projects without a Revision History section

Projects missing a Revision History section get NULL `start_date`,
`completion_date`, `revision_depth`. They're excluded from revision-depth
views and from the sophistication composite (flagged `too_early=True`).
No attempt to backfill from filesystem mtimes.

### B4. Phase 2b extraction is cost-gated

The universal extractor populates `entity_mentions` and `drift_candidates`
tables via LLM calls. For a corpus of ~50 projects with ~1200 extractable
sections, expect on the order of 5M tokens and 40–60 minutes wall-clock
with threaded execution (8 workers by default; configurable via
`EXTRACT_CONCURRENCY`). Serial execution is ~6x slower.

**Operational ordering constraint:** sophistication scoring and
research-line detection must run AFTER the extraction pass. If they run
first, the breadth axis stays at zero for every project despite successful
extraction. The scan orchestrator enforces the correct order; custom
pipelines should too.

Until Phase 2b runs, the following dashboard panels return zero or hide:

- top organisms / methods / databases by mention
- question-type portfolio
- conclusions-per-project
- conclusion-subject triangulation
- drift-candidates summary
- breadth axis of the sophistication composite (zeros; flagged via
  `partial_phase_2b=True`)

**Mitigation:** dashboard shows explicit "awaiting Phase 2b extraction"
banners on affected panels rather than misleading zeros.

### B5. REVIEW.md is auto-generated by a single LLM reviewer

BERIL's review infrastructure generates REVIEW.md files automatically via
a single configured reviewer model. Aggregate metrics derived from REVIEW
§Suggestions content reflect one model's consistent critique style, not
distributed expert opinion. The reviewer model signature is tracked
per-review, so "when did the reviewer change" is visible.

### B6. Single-user sandbox environment

The atlas is generally developed and tested on one operator's corpus at a
time. Multi-tenant deployment behavior (concurrent writes, cross-tenant
citation) is not exercised in the spike-level tests. Behavior in
multi-tenant deployments is expected to work via the `/submit` cross-install
sync but has not been benchmarked end-to-end.

## C · Entity extraction caveats

### C1. LLM extraction is non-deterministic across model versions

Same prose, different Claude Sonnet version, may produce different entity
sets or different canonical suggestions. The extraction cache is keyed by
`(content_hash, prompt_version, vocab_version, model_id)` so stored runs
are reproducible; a model upgrade forces re-extraction (visible as a
cache-invalidation event in the drift report).

### C2. Vocab coverage is seeded, not exhaustive

The shipped v1 vocabularies (organisms, methods, databases, journals,
functions, question-types) were hand-curated from sampling representative
projects during BERIL's bring-up. Real extraction reveals entities missing
from the seed; these surface in `drift_candidates` and feed the vocab
growth loop. The dashboard explicitly shows "proposed:..." canonical IDs
for unresolved mentions so reviewers can distinguish vocab-matched from
LLM-only. Users can add `.local.yaml` overlays in `vocab-local/` for
deployment-specific terms (see `vocab-reference.md`).

### C3. Author name extraction has markdown-noise artifacts

Author entries in the canonical `Authors` section of BERIL projects
sometimes include markdown artifacts or affiliation text that bleeds into
the `canonical_name` field. ORCID IDs are used as the canonical
identifier; the `canonical_name` field carries cosmetic noise. Author-graph
queries use `author_id` (ORCID-keyed) so analytics are unaffected; display
layer shows the noisy name verbatim for audit.

### C4. Inline citation extraction is partial

PMIDs and DOIs are extracted via regex. Author-year narrative citations
are extracted heuristically. `references.md` files are the richest citation
source but not all projects ship one. The citation graph is therefore
incomplete for projects without `references.md`.

### C5. SQL database schemas not introspected

The atlas knows *that* projects query specific BERDL databases (e.g.,
`kescience.fitnessbrowser`) but not *which columns* or what data-type each
column holds. Characterizing projects by "level of data touched" (raw
sequences vs. derived fitness) is not possible without BERDL schema
introspection. This is a known gap, flagged in `design-note` §12 wish list.

## D · Composite-score caveats

### D1. Sophistication is a corpus-relative z-score, not an absolute

Z-scoring against the current corpus means "a depth of +2.0" means "2σ
above the corpus mean," not an absolute measure. As new projects arrive,
the mean and std both shift; past scores become non-comparable. Every
sophistication export carries the run's `corpus_median` as an anchor.

### D2. Breadth axis requires Phase 2b to be meaningful

Without entity extraction, breadth = 0 for every project → z-score = 0
→ axis silent. Dashboard explicitly flags this via `partial_phase_2b=True`;
the scatter plots show a "breadth not yet populated" warning rather than
rendering a zero line.

### D3. Depth may correlate with time spent on the project, not "quality"

Depth rewards iteration (revisions, notebooks, byte volume). A project
that got it right in v1 scores lower on depth than one that iterated 5
times. This isn't wrong — "deep iteration" IS a kind of sophistication —
but confuses "mature" with "well-executed."

### D4. Influence accrues over time; depth is frozen at completion

New projects have near-zero influence by construction (no later projects
exist yet to cite them). Older projects have time to accumulate influence.
The "depth now vs. influence later" signal only works with a 4+ week lag.

### D4b. Influence / integration are CROSS-AUTHOR only

The composite axes are computed on cross-author edges only. A naive
in-degree/out-degree would conflate self-follow-on (one author citing
their own prior work) with true cross-author amplification, which in a
single-power-user corpus (see A2) would overstate influence.

- **Self-edges** — src and dst share ≥1 author. A separate
  `self_in_degree`/`self_out_degree` pair tracks these (the "deep-diver"
  pattern).
- **Cross-edges** — src and dst have disjoint author sets. These feed the
  influence (in) and integration (out) axes.

**What this means for readers:** influence and integration measure
cross-person knowledge amplification. Self-follow-on is reported
separately as its own signal (deep iteration by the same author). Both
are valid indicators but of different things.

### D5. Integration can be gamed

"Out-degree" rewards citing many prior projects. A project that cites 15
priors has higher integration than one that cites 2, even if the 2 are
deeper engagement. The dashboard reports out-degree distribution (not
just top rank) so readers can calibrate.

## E · Graph-metric caveats

### E1. Reuse graph edges are "declared" only (strictest tier)

An edge requires a strict backticked folder-name reference in a canonical
doc. Weaker signals (topical overlap, shared citations, shared datasets)
are NOT in the declared graph. This is deliberately conservative; the
dashboard may understate true amplification.

### E2. Transitive reach stops at 2 hops

We compute in-degree and 2-hop-in-degree. 3+ hops are not currently
computed. At typical corpus densities, this probably captures most
amplification paths, but very-deep propagation is invisible.

### E3. Research-line detection uses a specific heuristic

Research lines are weakly-connected components in the declared-citation
graph **plus** topic-overlap edges. A topic-overlap edge is added between
two projects when the cosine similarity of their (organism, method)
vocab-matched canonical sets is ≥0.5. The 0.5 threshold is tunable and
was empirically chosen during spike-scale validation — captures obvious
thematic pairs without connecting projects that share only extremely
common canonicals. Lines with ≥5 members are sharded into thematic
sub-clusters via Louvain community detection (resolution 1.5) on the
combined citation+topic graph restricted to line members; sub-cluster
edge weighting is `citation=1.0`, `topic=cosine`.

Known limitations: (1) `proposed:` canonicals are excluded from the
similarity signature — an unvoted drift candidate shouldn't create
line-level topology; (2) topic-edge computation depends on Phase 2b
having run — a Phase-1-only warehouse has `topic_edge_count=0` for every
line and no sub-clusters; (3) the 0.5 threshold and 1.5 resolution are
tuning knobs (see `research_lines.detect_research_lines` kwargs). Pre-2b
behavior is recoverable via `include_topic_edges=False`.

## F · Visualization caveats

### F1. Reuse network layout

Current layout is spring-force via NetworkX on the connected subgraph with
a circular fallback if NetworkX is unavailable. Readers should not infer
cluster structure purely from visual proximity when the network is sparse.

### F2. Time-series panels at small N are noisy

At corpus scales in the 50-project range, any chart binned by week has
~5–10 data points per bucket. Trendlines should display confidence
intervals where shown; readers should not over-interpret short-run
patterns.

### F3. Small-multiple charts reuse color scales

Colors are consistent across related panels (e.g., methods, domains,
modes) but users may miss this without a key. Consider a legend overlay
or a key-colors panel.

## G · Framing and interpretation caveats

### G1. Activity metrics ≠ outcome metrics

The atlas measures activity (projects, revisions, citations, notebooks).
It does not measure outcomes (time-to-insight, error rate, user
frustration, finding correctness). Any claim that BERIL "accelerated"
science must be tempered; "BERIL produced X projects with Y cross-
references" is honest.

### G2. External scientific impact not tracked

Citations of BERIL-produced findings by non-BERIL publications are not
tracked. The corpus is a closed universe for all metrics.

### G3. Finding correctness not validated

Claims extracted from REPORT §Key Findings are captured with source
quotes. The atlas asserts the claim was *made*; it does not assert the
claim is *correct*.

### G4. "Novelty" is intra-corpus only

First-in-BERIL timelines are computed. First-in-literature (vs. PubMed,
etc.) is explicitly out of scope per the architectural decision in
`design-note` §2.

## H · Operational caveats

### H1. Every dashboard run is timestamped but not versioned

Past dashboards live where they were written; comparing "today vs. last
week" requires keeping the prior HTML. No automatic diff.

### H2. Apply-drift mechanism not yet built

`drift-review.md` is generated; user can annotate; re-running does not
yet parse user decisions and bump vocabs automatically. A later phase
will close this loop. Users can manually promote drift candidates into
`vocab-local/` or upstream via the contribution flow (see
`CONTRIBUTION.md` in the package repo).

### H3. L6 recommendations are LLM-synthesized seed hypotheses

The L6 synthesis engine produces 5–8 next-direction recommendations with
evidence trace per rec, written to the `recommendations` table. **They
are NOT validated findings.** Each rec cites specific warehouse rows
(entities + line_ids + source_panel) but the synthesis itself is
unverified. At small N with a dominant author, the LLM has narrow ground
truth to draw from. Treat as starting points for human investigation;
validate with a domain expert before acting. Low-priority and
low-plausibility recommendations warrant extra caution.
