v0.3.14: mark_configured idempotency hotfix (append marker lines if absent)

Hotfix on top of v0.3.13. Closes a configure-time crash hit on Adam's
spike/beril-extended hub on 2026-05-05.

## What broke

A user's `.env` had `ACTIVE_PROVIDER=cborg` + `CBORG_API_KEY=<real>`
set (e.g., from a partial prior setup that left ACTIVE_PROVIDER set
externally, or another tool populated it). The atlas marker stanza
(`BERIL_ATLAS_CONFIGURED_AT=` / `BERIL_ATLAS_CONFIGURED_VERSION=`)
was NOT in the file.

The configure flow correctly classified the state as
`keys-present-unverified` (per `config_status.py:109` — keys are
present, marker timestamp is empty), ran the smoke test (passed),
then called `mark_configured.run` to stamp the marker.

`mark_configured._update_line` was a regex-replace-only: when the
target line was physically absent, `subn` returned count=0 and
the function returned `(text, False)`. The caller then errored:

    Error: BERIL_ATLAS_CONFIGURED_AT/BERIL_ATLAS_CONFIGURED_VERSION
    not found in .../.env. The atlas template hasn't been appended
    yet — run `/beril-atlas-configure` to add it.

The error message blamed the user (run configure!) but configure
HAD just run. The fix needed to be in mark_configured itself, not
the configure flow.

## What's fixed

`mark_configured._upsert_marker_lines` is the new canonical entry
point: replace-if-present, append-if-absent. When the .env has no
atlas template block (i.e., neither the v0.x configuration comment
header nor the v0.3.14 marker-block header is present), a one-time
self-documenting marker-block comment header is added before the
appended lines so the stanza is readable in raw text.

Idempotent across repeated invocations: no duplicate lines, no
duplicate headers.

The state machine in `config_status` is unchanged — its
classification of the user's situation as `keys-present-unverified`
was correct; the bug was in the mark step's assumption about file
shape. Future analogous shapes (additional marker lines added in
v0.4, partial template cleanup by the user, third-party
ACTIVE_PROVIDER assignment) all flow through the same upsert path.

The pre-existing `_update_line` helper is kept for backwards
compatibility with any code path calling it directly, and the new
upsert composes it.

## Tests (6 new, 159/159 pass)

`tests/unit/test_v0314_mark_configured_idempotent.py`:

- **test_appends_marker_lines_when_absent** — exact reproducer from
  Adam's hub: pre-existing ACTIVE_PROVIDER + key, no marker stanza.
  Pre-fix: exit 2 with "not found." Post-fix: exit 0, both lines
  appended with valid ISO timestamp + current version.
- **test_appends_with_header_when_no_atlas_block_present** —
  marker-block comment header precedes appended lines for
  self-documenting raw-file readability.
- **test_replaces_existing_lines_when_present** — pre-v0.3.14
  behavior preserved when marker lines exist (regex replace, no
  append, no duplicate header). Verifies the well-formed-template
  case isn't disturbed.
- **test_only_one_marker_line_present_appends_the_other** —
  partial-template edge case where AT exists but VERSION doesn't.
  Replace AT, append VERSION; exactly one of each.
- **test_idempotent_double_invocation** — running twice produces no
  duplicate lines and no duplicate header.
- **test_errors_when_env_file_missing** — pre-existing error path
  for missing-.env still returns exit 1.

## Backward compatibility

- Existing well-formed `.env` files unchanged: `mark_configured` runs
  exactly as before (regex-replace existing lines).
- Existing partial `.env` files (the failure case): `mark_configured`
  now succeeds where it previously errored.
- No schema changes. No CLI surface changes. No state machine
  changes. State classification semantics in `config_status`
  unchanged.
- Old workaround (`cat >> .env <<EOF ... EOF`) still works on v0.3.14
  but is no longer necessary.

## Cross-skill implications

This is an atlas-only bug. Adversarial / paper-writer /
presentation-maker delegate provider auth to the `claude` / `codex`
CLIs and don't track their own marker state in `.env`. The
architectural asymmetry was discussed in the 2026-05-05 cross-skill
thread and is correct: atlas's marker pattern is justified by its
standalone-provider use case, but the skill must handle .envs in
arbitrary partial-prior-setup states. v0.3.14 is the resilience
patch the marker pattern requires to be safe.
